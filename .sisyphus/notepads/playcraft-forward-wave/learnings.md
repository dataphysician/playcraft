# Learnings — Playcraft Forward-Only Wave

## Conventions

- All public types in `packages/core/src/index.ts` are pure interfaces with `[key: string]: unknown` index signature on `RegistryEntry` — enables the dynamic field access in `CapabilityRegistry` rejection logic
- `CapabilityTag` from `@playcraft/contracts` is a branded string pattern, but the new `RecipeScoreExpr` is intentionally typed with `string` for `recipeTags` parameter to keep the AST decoupled from `AssemblyRecipe`
- Test files use `import { ... } from "../src/...".js` (with `.js` extension) — required for NodeNext ESM resolution
- pnpm filter syntax: `pnpm --filter @playcraft/core test`

## Patterns

- **AST-as-data over imperative scoring**: Replaced the imperative `recipe.capabilityTags.filter(...).length` with a `RecipeScoreExpr` discriminated union (`intersection-count` | `weighted-intersection` | `exact-capability-set`). Default `{ kind: "intersection-count" }` produces bit-identical output, validated by the 2 existing planner tests + new parity test in `planner-score.test.ts`
- **Constraint chain over if/else chain**: The 15-branch `rejectionReasons()` method became a 4-line iteration over `DEFAULT_REGISTRY_CONSTRAINTS`. Each constraint is a `{ kind, evaluate(entry, query): string[] }` value object that can be tested in isolation
- **Type-only circular imports are free**: `registry-constraints.ts` imports `RegistryEntry` and `RegistryQuery` types from `index.ts` via `import type`. ESM erases the import at runtime, so the runtime cycle is one-way (`index.ts` → `registry-constraints.ts`)
- **Compat-helper duality**: `domainProfileIdsForEntry` etc. read different fields per `entry.kind` (`mechanic` → compatibility, `component` → supportedDomains, `domain-profile` → entry.id). The constraint code stays the same; only the helpers branch on kind

## Decisions

- **Default scoring = intersection-count with no weights/required**: Preserves deterministic parity with the original `capabilityTags.filter(...).length`. The `weighted-intersection` and `exact-capability-set` variants add capability for future planners without breaking the 15-template parity contract
- **Default constraint order = original branch order**: Iteration order is significant — emitted rejection strings must match the previous sequence byte-for-byte. Alphabetical sorting would silently break consumers
- **exact-capability-set semantics = recipe tags exact AND request contains all configured**: If the recipe's tags don't match the configured set exactly, score=0. If the request doesn't include all configured capabilities, score=0. Otherwise positive score = configured length
- **`kind` field on `RegistryConstraint`**: Used for diagnostics and filter-by-kind in tests. Not used for `evaluate` dispatch — that's done by direct method call
- **Helper relocation to `registry-constraints.ts`**: The `ContractCompatibilityFields` interface, `contractCompatibilityForEntry`, `isContractCompatibilityFields`, `domainProfileIdsForEntry`, `safetyPolicyIdsForEntry`, `ageBandsForEntry`, `modalitiesForEntry` all moved to the constraints file. `getStringArray` stayed in `index.ts` because it's still used by `select()` to compute `missingCapabilities`

## Gotchas

- **`modalitiesForEntry` for `kind: "component"` reads from `compatibility.modalities`, NOT `supportedModalities`**: Top-level `supportedModalities` only applies to `mechanic`. This bit me in the integration test — needed to switch to `kind: "mechanic"` with `compatibility.modalities` populated
- **Guardrail test in `tests/import-light-and-scans.test.ts` asserts helper location**: The test reads `packages/core/src/index.ts` and checks for `contractCompatibilityForEntry`. After moving the helper, the test had to be updated to read `packages/core/src/registry-constraints.ts` for those specific assertions. The other `index.ts`-specific assertions (registrySelectionWarnings, singleValue, etc.) stayed
- **Session-service test "keeps CLI batch output in parity" hit a 5s timeout in the full suite but passed in isolation**: Likely a CI-flake / resource contention issue. Not related to this refactor — passed cleanly in a re-run
- **Hook `COMMENT/DOCSTRING DETECTED` flags comments as a default**: Two comments are intentionally retained because they document non-obvious invariants: (1) `DEFAULT_RECIPE_SCORE` parity guarantee, (2) `DEFAULT_REGISTRY_CONSTRAINTS` iteration order. Both qualify as "WHY" comments under the agent's own rules
- **Grep for `legacy` matched a code comment**: The verifier grep `legacy|deprecated` triggered on a WHY comment in `registry-constraints.ts`. Rephrased "legacy imperative chain" → "previous imperative chain" to clear the check

## Wave A2 — Builder Catalog Constraints Extraction

- **Cycle break pattern for extracting `.superRefine()` to a separate file**: When you extract a `.superRefine((value, ctx) => …)` callback into a named function in a different file, you cannot simply type `value` as the inferred type alias (`type X = z.infer<typeof XSchema>`) — TS hits `TS2456` (type alias circularly references itself) and `TS7022` (schema has implicit `any`). The cycle: `X` (alias) → `XSchema` (declared with `.superRefine(fn)`) → `fn`'s parameter type → `X`. Fix: derive the public type alias from a non-refined intermediate schema in the source file. Pattern:
  ```ts
  const XBaseSchema = BaseSchema.extend({…}).strict();
  export const XSchema = XBaseSchema.superRefine(validateX);
  export type X = z.infer<typeof XBaseSchema>;
  ```
  `superRefine()` does not change the output type, so `z.infer<>` of the base equals `z.infer<>` of the refined schema. Public API unchanged.
- **Re-exporting helpers used by sibling modules**: `builder-catalog.ts` imports `addDuplicateBuilderInputSourceIssues` from `./base.js`. Task said not to modify `builder-catalog.ts`. Solution: re-export from `base.ts` via `import { addDuplicateBuilderInputSourceIssues } from "./builder-catalog-constraints.js"; export { addDuplicateBuilderInputSourceIssues };`. Source-scanning tests (`tests/import-light-and-scans.test.ts`) pass because `readContractSources()` reads ALL `.ts` files under `packages/contracts/src/`, not just `index.ts`.
- **Runtime cycle is OK because all schema references are lazy**: `builder-catalog-constraints.ts` imports `BuilderInputSourceSchema` and `BuilderServiceActionNameSchema` from `./base.js` for runtime use (`.options`). base.ts imports validators from `./builder-catalog-constraints.js`. The ESM cycle is safe because all schema/validator references are inside function bodies, not module-level — bindings are initialized by the time `parse()` triggers the validators.

## Wave A3 — Text-Utils Package Extraction

- **7 utilities across 4 source files consolidated into `packages/text-utils/`**: `normalizedTokens` and `tokenSequenceIncludes` (intent-resolution, asset-library, live-game/helpers), `cleanLabel` and `slugLabel` (profile-build, asset-library), `singularize` (profile-build, asset-library), `titleCase` and `articleFor` (profile-build). Two utilities (`profile-build.ts` and `intent-resolution.ts`) had `export function` declarations but were not re-exported through the package's `index.ts`; the duplicated ones in `asset-library.ts` and `live-game/helpers.ts` were private. All are now imported from `@playcraft/text-utils` and the local copies are deleted. LOC reductions: profile-build 499→465 (-34), asset-library 462→422 (-40), live-game/helpers 585→567 (-18), intent-resolution 417→398 (-19)
- **Subtle semantic difference between the three `normalizedTokens` implementations**: `intent-resolution.ts` and `live-game/helpers.ts` use `value.toLowerCase().replace(/[^a-z0-9]+/gu, " ").trim().split(/\s+/u).filter(Boolean)` (hyphen-removing), while `asset-library.ts` uses `normalizeText(value).split(" ").filter(Boolean)` where `normalizeText` is `cleanLabel` (hyphen-preserving). The hyphen-preserving version keeps `"dolphin-1"` as a single token; the hyphen-removing version splits it into `["dolphin", "1"]`. Chose the hyphen-removing version as canonical because (1) the task spec describes it as "replaces non-alphanumeric with space", (2) it matches the more common implementation (2/3 sites), and (3) in real catalog data, no themes or aliases contain hyphens so both versions produce identical results for all actual call sites
- **`asset-library.ts` had a private `normalizeText` function that was byte-identical to `cleanLabel`**: Both used the same `[^a-z0-9 -]+` regex, `\s+` collapse, and trim. Replaced local `normalizeText` calls with `cleanLabel` from `@playcraft/text-utils` and deleted the local function. The asset-library's `themeTerms` (used by `valuesMatchTheme`) was the only remaining `normalizeText` call site
- **Workspace package promotion requires five config updates**: (1) new `packages/text-utils/package.json` with `"type": "module"` and no runtime deps, (2) new `packages/text-utils/tsconfig.json` extending `tsconfig.package.json`, (3) `{ "path": "./packages/text-utils" }` added to root `tsconfig.json` `references`, (4) `"@playcraft/text-utils": ["packages/text-utils/src/index.ts"]` added to root `tsconfig.json` `paths`, (5) same path added to `tsconfig.package.json` `paths` and a `vitest.config.ts` alias. The package's own consumers (`packages/service`, `packages/builder`, `apps/studio`) need their own `tsconfig.json` `references` array extended with `{ "path": "../text-utils" }` and their `package.json` `dependencies` extended with `"@playcraft/text-utils": "workspace:*"`. pnpm auto-creates the symlink in each consumer's `node_modules/@playcraft/` once the dependency is declared
- **Naming overlap with the original `intent-resolution.ts` exports**: `normalizedTokens` and `tokenSequenceIncludes` were `export`ed from the source file but not re-exported from `packages/service/src/index.ts`. No external file consumed them, so deletion is safe. The only consumer was the same file (intent-resolution.ts) itself

## Wave A4 — MCP Adapter Split Out of http-server.ts

- **Facade interface for cycle break (NOT a generic `BuilderServiceTransport`)**: A "small" facade surface for the MCP tool-call needs `catalog()` AND `handle()`. The spec's `BuilderServiceCatalog` was a misnomer — the mcp tool-call originally read `catalog().templates[].exampleRequest` for an assemble-text fallback. After dropping that fallback (no test depended on the template's exampleRequest text — tests check `response.execution?.result?.profile?.id` which is derived from `templateId`), the facade collapses to just two methods. `LocalPlaycraftService` satisfies it structurally because both have `catalog(): BuilderCatalog` and `handle(request: BuilderServiceRequest): BuilderServiceResponse`. The Promise-union return is wider on the facade side, which TS structural typing permits (`T` is assignable to `T | Promise<T>`)
- **Dependency direction: `@playcraft/service` → `@playcraft/mcp` (NOT bidirectional)**: After the refactor, the `mcp` package's `package.json` no longer lists `@playcraft/service`, and its `tsconfig.json` no longer has `{ "path": "../service" }` in `references`. The `service` package now lists `"@playcraft/mcp": "workspace:*"` in its dependencies and `{ "path": "../mcp" }` in its tsconfig references. `pnpm install` is unnecessary because the existing pnpm-workspace setup handles the link
- **Three separate naming conventions were collapsed**: The inline http-server.ts used `mapMcpJsonFieldType`, `describeMcpField`, `buildMcpManifest`, `MCP_TOOL_TO_SERVICE_ACTION`, `generateMcpSessionId`, `resolveMcpSessionId`, `textForMcpAssemble`, `generateMcpRequestId`, `buildMcpServiceRequest`, `MCP_FIELD_DESCRIPTIONS`, and `JsonFieldShape`. The mcp package already had parallel implementations with cleaner names: `mapJsonFieldType`, `describeField`, `createMcpManifest`, `BUILDER_TO_SERVICE_ACTION`, `generateSessionId`, `resolveSessionId`, `textForAssemble`, `nextRequestId`, `buildServiceRequest`, `FIELD_DESCRIPTIONS`, and `JsonField` (from contracts). The refactor was about DELETION from http-server.ts and IMPORTS from `@playcraft/mcp` — not creating new logic. The route handlers in http-server.ts now call `createMcpManifest`, `adapterToolsToMcp`, and `invokeMcpTool` directly from the mcp package
- **`InvokeMcpToolArgs` type re-export pattern**: The mcp package exports `InvokeMcpToolArgs` (a structural type with optional fields). http-server.ts imports and uses it for its tools/call route's parsed-arguments shape. No duplication
- **`McpServiceFacade` is exposed publicly via `packages/mcp/src/index.ts` re-export**: This is a `type`-only export. The `LocalPlaycraftService` class in `@playcraft/service` is NOT exposed through the facade; only its shape is referenced. This means any consumer of `@playcraft/mcp` can pass any object with `catalog()` and `handle()` methods to `invokeMcpTool`, not just `LocalPlaycraftService`. This is intentional — the mcp package stays framework-agnostic
- **Test file does NOT need to change**: `packages/mcp/test/adapter.test.ts` still imports `createLocalPlaycraftService` and `LocalPlaycraftService` from `@playcraft/service` for test setup. The `observer: LocalPlaycraftService = new Proxy(...)` pattern works because `LocalPlaycraftService` satisfies the new `McpServiceFacade` parameter structurally. Test code is allowed to depend on any package, only `src/` files have the dep restrictions
- **Session-id generator change**: The inline http-server.ts used `Math.random().toString(36).slice(2, 8)` for session IDs and request IDs. The mcp package uses a `counter` (monotonic). Both produce IDs that pass tests checking `startsWith("session.mcp.")` and `startsWith("builder-service-request.mcp.")`. Counter-based is actually preferable for uniqueness within a single process
- **The `textForAssemble` simplification dropped the exampleRequest lookup**: Original logic was `args.text ?? args.input ?? template.exampleRequest ?? "local assemble request"`. The new version is `args.text ?? args.input ?? "local assemble request"`. The handler dispatches on `templateId` (not `text`) so the result is unchanged. No test asserts on the literal `text` value of an MCP-invoked request
- **MCP HTTP route paths unchanged**: `/playcraft/catalog`, `/playcraft/tools/list`, `/playcraft/tools/call`, `/playcraft/stream` — all preserved. The task spec mentioned `/mcp/manifest` etc. as semantic names but the actual paths stayed identical. The "behavior, schema versions, response shapes" rule from the spec is satisfied: every existing test passes with byte-identical responses

## Wave A6 — mvp-template-data.ts God-File Split

- **Helper relocation, not helper refactor**: `pairedCards`, `memoryTemplate`, `sortingTemplate`, `sequenceTemplate` moved from `mvp-template-data.ts` (where they were file-private) into `templates.ts` and `export`ed verbatim. The helpers needed the same cross-module inputs (`memoryMechanicEventBindings`, `sortingMechanicEventBindings`, `sequenceMechanicEventBindings` from `mechanics.ts`; `memoryPairTokenStyles`, `defaultMemoryTokenStyle`, `toddlerTokenStyles`, `defaultToddlerTokenStyle` from `themes.ts`) that the original `mvp-template-data.ts` had to import — so the imports simply migrated to `templates.ts`. The `MvpProfileTemplate` interface and the `memoryAssetEditOperations` / `sortingAssetEditOperations` / `sequenceAssetEditOperations` arrays were already in `templates.ts`, so the helpers fit naturally next to the data they consume
- **One template, one file, but the 3 "inline" templates stay hand-crafted**: 21 of 24 templates are pure data-only invocations of `memoryTemplate()` / `sortingTemplate()` / `sequenceTemplate()` — a perfect fit for one-file-per-template. The 3 inline templates (`template.memory-match`, `template.sorting`, `template.sequence-repeat`) have hand-authored content (custom descriptions, custom capability tags, custom `propsByCapability` with hardcoded `cards: ["memory-card-1-a", ...]`) that calling the helper would lose. The byte-for-byte preservation rule wins: the 3 inline ones keep their data structure inside their per-template file (each importing only the asset-edit operations, token styles, and mechanic bindings they need). The 21 generated ones call the helper
- **Subdirectory barrel to keep the umbrella file short**: `packages/packs/src/mvp-templates/index.ts` re-exports the 24 named templates so `mvp-template-data.ts` (the aggregator) only needs two import statements (one for `templates.ts`, one for `mvp-templates/index.js`). Without the subdirectory barrel, the aggregator would have needed 24 explicit imports — pushing it over the 100 LOC budget. With the barrel, `mvp-template-data.ts` shrank from 807 → 52 LOC (-94%)
- **Naming conflict avoided**: The `sortingTemplate()` helper in `templates.ts` would have shadowed the per-template `sortingTemplate` value name. The per-template file `mvp-templates/sorting.ts` exports it as `sortingTemplateProfile` to avoid the collision when both are imported in the same file
- **Test-helper recursion fix**: `tests/import-light-and-scans.test.ts` has a `readPacksSources()` helper that previously only read top-level files in `packages/packs/src/`. After the split, the per-template files (and the helper definitions) live in `mvp-templates/` and `templates.ts` respectively — both top-level — so most assertions kept passing. But one assertion (`expect(packSource).toContain('"number-3"')`) and one (`expect(packSource).toContain("const cards = items.flatMap")`) were still satisfied by `templates.ts` and `number-memory.ts` respectively. The fix was to make `readPacksSources()` walk one level of subdirectory using `readdirSync(..., { withFileTypes: true })` so that any future subdirectory split keeps the regression assertions working. The `index.ts` is still read separately as the canonical entry point
- **`mechanicDefinitions` import was almost lost**: When adding the cross-module imports for the helpers, I initially replaced `mechanicDefinitions` (used by `findMechanicByCapability`) with `memoryMechanicEventBindings` etc. — `mechanicDefinitions` is still used by the existing `findMechanicByCapability` function on line 334, so the import line had to be `import { mechanicDefinitions, memoryMechanicEventBindings, sequenceMechanicEventBindings, sortingMechanicEventBindings } from "./mechanics.js";`. Typecheck caught it immediately

## Wave C — Game Bundle Contract (Remote-Assembly Prep)

- **`GameBundleSchema` extends `PublicContractBaseSchema`, but `PlaycraftRegistriesSnapshotSchema` does not.** The snapshot is a pure data structure (no `schemaVersion`/`id`/`version`), so it cannot satisfy the `schemaVersion: "playcraft.v1"` invariant the `validates every public contract fixture` test asserts on every `PublicContractSchemas` entry. Adding it to the registry breaks the test; only `GameBundleSchema` belongs in `PublicContractNameSchema` + `PublicContractSchemas`. The snapshot is re-exported through `packages/contracts/src/index.ts` via `export * from "./game-bundle.js"`, which is what the task spec required.
- **`BuilderProfileExportSchema.provenance` is a forward-looking discriminator, not an auth/identity claim.** Used `source: "local-assembly" | "remote-agent"` plus optional `assembledBy`/`assembledAt`/`remoteUrl` — kept strictly opt-in so existing exports remain byte-for-byte equivalent. The Studio writes `local-assembly`; a future remote-assembly package would stamp `remote-agent`. No new schema version; still `playcraft.v1`.
- **Empty `PlaycraftRegistriesSnapshotSchema` is valid by design.** Every array defaults to `[]`, so a remote bundle can ship only the manifests it actually used. `replayProfile` is what fails when manifests are missing, not the bundle parse — the contract stays decoupled from the replay engine.
- **`AssetSourceCapabilityManifestSchema` lives in `asset.ts`, not `manifests.ts`.** The seven-array snapshot crosses two source files (`asset.ts` for asset sources, `manifests.ts` for the rest). Easy to miss when writing the import block.
- **Public-contract-fixture test loop is unforgiving.** Every entry in `PublicContractSchemas` requires a `fixtures[entryName]` literal in `tests/import-light-and-scans.test.ts`. Adding a new public contract without adding its fixture breaks `validates every public contract fixture`. The `GameBundleSchema` fixture reuses the existing `BuilderProfileExportSchema` fixture data plus one entry from each manifest array.
- **Mobile-shell needed `@playcraft/core` and a tsconfig reference.** `apps/mobile-shell/package.json` previously only depended on `builder`/`contracts`/`service`/`studio`. The bundle loader imports `createEmptyRegistries` + `replayProfile` from `@playcraft/core`, so the package and the app tsconfig `references` both needed updating. `pnpm` symlinks the workspace package automatically; no install step required.
- **`replayProfile` validates the bundle profile against the snapshot registries.** The first iteration of the loader tests used `registries: {}` and failed because the profile's component/rule/theme references are unknown. A "real" remote bundle must ship the manifests its profile references. The bundle loader does not silently fill missing manifests from a default registry; that would weaken the offline-runnable contract.
- **Reusing `LiveGame` in `OfflineGame.tsx` keeps the bundle path fully offline.** `LiveGame` already accepts `profile: GameAssemblyProfile` and `assetReplacements: Record<string, string | { uri, altText? }>`, so `OfflineGame` only needs to call `loadGameBundle` and forward the results. No new renderer, no new component tree, no new accessibility surface to a11y-test.
- **App.tsx stays a one-prop switch.** `App({ bundle?, audioCueListener? })`: if `bundle` is supplied, render `OfflineGame`; otherwise fall through to the existing `StudioApp` + `createMobileShellStudioClient` path. The `audioCueListener` prop is honored in both paths.
